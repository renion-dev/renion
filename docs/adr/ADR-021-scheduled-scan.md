# ADR-021: Автоматичний запуск сканування за розкладом

**Дата:** 2026-07-23  
**Статус:** Прийнято  

## Контекст
На поточному етапі сканування запускається тільки вручну через веб-інтерфейс або CLI. Для повноцінної роботи Opportunity Hunter необхідно автоматизувати цей процес, щоб нові гіпотези генерувалися регулярно без участі користувача.

## Рішення

### Підхід
- Використовувати `APScheduler` для запуску сканування за розкладом у фоновому режимі.
- Планувальник працює в тому ж процесі FastAPI (без окремого воркера).
- Розклад налаштовується через змінну середовища `SCAN_SCHEDULE` (у форматі cron або інтервал у хвилинах).
- За замовчуванням — запуск щодня о 8:00 (UTC).

### Технічна реалізація
- Додати залежність `APScheduler`.
- Створити модуль `src/infrastructure/scheduler.py` з класом `ScanScheduler`.
- Ініціалізувати планувальник у `web.py` при старті.
- Використовувати `AsyncIOScheduler` для асинхронного виконання.
- Запобігати одночасному запуску кількох сканувань (перевіряти статус `ScanJob`).

### Конфігурація
- `SCAN_SCHEDULE` — cron-вираз (наприклад, `0 8 * * *`) або інтервал у хвилинах (наприклад, `1440` для щоденного).
- Якщо не вказано — сканування не запускається автоматично.

### Альтернативи
- Використання зовнішнього cron-сервісу (наприклад, cron-job.org) — додаткова залежність.
- Окремий Docker-контейнер для планувальника — ускладнює архітектуру.

## Наслідки
- Користувачі отримують свіжі гіпотези без ручних дій.
- Збільшується навантаження на API (Groq, Stripe) — потрібно контролювати ліміти.
- Потрібно додати логування для відстеження автоматичних сканувань.

## Майбутні плани
- Додати можливість налаштовувати розклад через веб-інтерфейс.
- Додати сповіщення (email) про нові гіпотези.
- Інтеграція з Telegram-ботом для повідомлень.
