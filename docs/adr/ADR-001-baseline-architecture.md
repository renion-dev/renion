# ADR-001: Базові архітектурні рішення AION

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
Ми створюємо операційне середовище для автономних цифрових систем (AION).  
Основні вимоги:
- Локальне виконання (Local First)
- Подієво-орієнтована архітектура (Event-Driven)
- Можливість заміни будь-якого компонента
- Ядро не залежить від зовнішніх сервісів

## Рішення

### 1. Мова програмування: Python 3.10+
- **Чому:** Швидка розробка, велика екосистема, підтримка asyncio.
- **Альтернативи:** Node.js, Go, Rust — обрано Python через швидкість прототипування.

### 2. База даних: SQLite + aiosqlite
- **Чому:** Локальне сховище, простота, транзакційність, підтримка JSON.
- **Альтернативи:** PostgreSQL (занадто важкий), Redis (не підходить для історії).
- **Компроміси:** Обмежена масштабованість, але легко замінити через абстракцію.

### 3. Архітектура проєкту: Clean Architecture (за RENE)
- **Структура:**
  - `src/domain/` — сутності (Object, Event) — жодних зовнішніх залежностей.
  - `src/application/` — use cases (OpportunityHunter) — координація об'єктів.
  - `src/infrastructure/` — Storage, EventBus, зовнішні API — залежать від domain.
  - `src/interfaces/` — CLI, пізніше REST/Telegram — точки входу.
- **Чому:** Розділення відповідальності, легкість заміни інфраструктури.

### 4. Модель даних: Універсальний Object + Event
- Кожен об'єкт має: `id`, `type`, `metadata`, `history` (список подій).
- Кожна зміна фіксується як `Event` з `payload` і зберігається в БД.
- **Чому:** Універсальність, повна історія, легке відстеження змін.

### 5. Обробка подій: Асинхронна шина (EventBus)
- Публікація → збереження в БД → черга → обробники.
- Підписка за типом події.
- **Чому:** Асинхронність, масштабованість, розв'язка компонентів.

## Альтернативи, що не пройшли
- Використання Redis як черги — зайва залежність для MVP.
- REST API замість локального виклику — оверінжиніринг на старті.
- Використання ORM (SQLAlchemy) — збільшує складність, достатньо raw SQL.

## Наслідки
- Можна швидко замінити SQLite на PostgreSQL, змінивши лише `storage.py`.
- Нова бізнес-логіка (інші агенти) додається як нові use cases і підписки на події.
- Тестування спрощується завдяки `:memory:` базі даних.

## Майбутні зміни (передбачення)
- Додавання черг (RabbitMQ/Kafka) при зростанні навантаження.
- Підключення LLM як зовнішнього сервісу через інфраструктурний шар.
- Розширення моделі об'єкта для підтримки версіонування.
