# ADR-014: Stripe Webhook

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
Поточний платіжний процесор (`PaymentProcessor`) використовує синхронне підтвердження платежів через `confirm_payment()`. Це працює для заглушки, але для реального Stripe необхідно обробляти асинхронні події (особливо для карток з 3D Secure, банківських переказів тощо). Webhook дозволяє Stripe повідомляти наш сервер про зміну статусу платежу.

## Рішення

### Підхід
- Створити ендпоінт `/webhook/stripe` у FastAPI додатку.
- Приймати POST-запити від Stripe з підписом (верифікувати через `STRIPE_WEBHOOK_SECRET`).
- Обробляти події:
  - `payment_intent.succeeded` → оновити статус платежу на `completed`, інвойс на `paid`.
  - `payment_intent.payment_failed` → оновити статус на `failed`.
  - `charge.refunded` → оновити статус на `refunded`.

### Технічна реалізація
- Використовувати `stripe.Webhook.construct_event()` для верифікації.
- Отримувати `payment_intent_id` з події, знаходити відповідний `Payment` у БД за `provider_reference`.
- Оновлювати статус платежу та інвойсу через методи `storage.save_payment()` та `storage.save_invoice()`.
- Повертати `200 OK` (Stripe очікує цей статус).

### Безпека
- Webhook секрет зберігається в `.env` (`STRIPE_WEBHOOK_SECRET`).
- Ендпоінт захищено від публічного доступу (тільки POST).
- У продакшні використовувати HTTPS.

### Альтернативи
- Polling (періодичне опитування Stripe API) — менш ефективно, більше запитів.
- Використання Stripe Dashboard для ручного підтвердження — не автоматизовано.

## Наслідки
- Потрібно налаштувати Webhook у Stripe Dashboard (у тестовому режимі).
- Необхідно додати публічний URL для прийому webhook (можна використовувати ngrok для локальної розробки).
- Webhook стає єдиним джерелом істини для статусів платежів.

## Майбутні плани
- Додати обробку інших типів подій (наприклад, `invoice.paid` для підписок).
- Логувати всі webhook-запити для аудиту.
- Додати retry-механізм (Stripe автоматично повторює запити, але ми можемо обробляти дублікати).
