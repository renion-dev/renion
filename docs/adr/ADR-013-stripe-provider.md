# ADR-013: StripeProvider (реальна оплата)

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
На поточному етапі платіжний шар AION використовує `SimulatedProvider` (заглушку). Для виходу в продакшн необхідно замінити його на реальний платіжний провайдер. Stripe є найбільш популярним та зручним рішенням для прийому платежів.

## Рішення

### Провайдер
- **Stripe** — глобальна платіжна платформа з підтримкою карток, Apple Pay, Google Pay та інших методів.

### Технічна реалізація
- Використовувати офіційний Python SDK: `stripe`.
- Створити клас `StripeProvider`, який імплементує інтерфейс `PaymentProvider`.
- Методи:
  - `create_payment(amount, currency, metadata) -> Payment`
  - `confirm_payment(payment_id) -> bool`
  - `get_payment_status(payment_id) -> str`
  - `refund_payment(payment_id) -> bool`
- Для асинхронності використовувати `asyncio.to_thread()` для викликів Stripe.

### Конфігурація
- API ключі через змінні середовища: `STRIPE_SECRET_KEY`, `STRIPE_PUBLISHABLE_KEY`.
- Webhook для підтвердження платежів (опціонально, на першому етапі можна використовувати синхронне підтвердження).

### Безпека
- Ключі не зберігаються в коді.
- Використовувати `python-dotenv` для завантаження з `.env`.

### Webhook (майбутнє)
- Додати ендпоінт `/webhook/stripe` для обробки подій від Stripe (payment_intent.succeeded, payment_intent.payment_failed).
- Це дозволить оновлювати статус платежів асинхронно.

## Альтернативи
- PayPal — складніша інтеграція, менш зручний API.
- Paddle — хороший для SaaS, але дорожчий.
- Lemon Squeezy — простий, але менш гнучкий.

## Наслідки
- Заглушку `SimulatedProvider` можна залишити для тестування (перемикати через конфігурацію).
- Для роботи з Stripe потрібен обліковий запис та API ключі.
- Потрібно додати залежність `stripe` у `pyproject.toml`.

## Майбутні плани
- Додати підписки (recurring payments) через Stripe Subscriptions.
- Додати webhook для автоматичного оновлення статусу платежів.
- Підтримка інших валют.
