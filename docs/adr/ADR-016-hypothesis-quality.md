# ADR-016: Покращення якості гіпотез (кластеризація + оцінка TAM/SAM)

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
На поточному етапі Opportunity Hunter генерує гіпотези на основі аналізу статей через LLM. Однак:
- Гіпотези часто дублюються (різні формулювання однієї проблеми).
- Немає кількісної оцінки потенціалу ринку (TAM, SAM, SOM).
- Немає пріоритизації — яка гіпотеза найбільш перспективна.

Це знижує цінність для користувача: він отримує багато гіпотез, але не знає, з якої почати.

## Рішення

### 1. Кластеризація проблем

**Підхід:**
- Використовувати embeddings (через Ollama або локальну модель) для перетворення тексту проблеми у вектор.
- Застосувати алгоритм кластеризації (K-Means або DBSCAN) для групування схожих проблем.
- Кожен кластер отримує:
  - Центроїд (узагальнений опис)
  - Кількість статей у кластері (частота)
  - Список ключових слів

**Технічна реалізація:**
- Модуль `src/application/clustering.py` з класом `HypothesisClusterer`.
- Використання `sentence-transformers` (або `ollama.embeddings`) для отримання векторів.
- Збереження кластерів у БД (тип `Cluster`).

### 2. Оцінка потенціалу (TAM/SAM/SOM)

**Підхід:**
- Використовувати LLM для оцінки:
  - **TAM** (Total Addressable Market) — загальний обсяг ринку.
  - **SAM** (Serviceable Addressable Market) — доступна частина.
  - **SOM** (Serviceable Obtainable Market) — реально досяжна частка.
- На основі частоти проблеми, кількості згадувань у різних джерелах, та контексту (наприклад, вакансії, GitHub Issues).
- Додати оцінку **"потенційний дохід"** (TAM * ціна * частка ринку).

**Технічна реалізація:**
- Додати метод `estimate_market_potential(hypothesis_data)` у `OpportunityAnalyzer`.
- Повертати структуру: `{"tam": "$X", "sam": "$Y", "som": "$Z", "confidence": 0.8}`.
- Зберігати у метаданих гіпотези.

### 3. Пріоритизація гіпотез

**Підхід:**
- Розраховувати **score** для кожної гіпотези на основі:
  - Частота проблеми (кількість статей/згадувань)
  - Потенційний дохід (TAM * ціна * 0.01)
  - Легкість реалізації MVP (оцінюється LLM)
  - Конкуренція (оцінюється LLM)
- Використовувати зважену суму для фінального рейтингу.

## Альтернативи
- Використання зовнішніх API для оцінки ринку (наприклад, Crunchbase) — потребує інтеграції та ключів.
- Ручна оцінка — не масштабується.

## Наслідки
- Користувач отримує не просто список гіпотез, а **ранжований список** з оцінкою потенціалу.
- Підвищується цінність продукту.
- Додаткове навантаження на LLM (для оцінки TAM/SAM) та на обчислення (для embeddings).

## Майбутні плани
- Інтеграція з зовнішніми API для точнішої оцінки ринку.
- Додавання графіків та візуалізації у веб-інтерфейсі.
- Автоматичне оновлення оцінок при зміні даних.
