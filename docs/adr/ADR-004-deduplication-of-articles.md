# ADR-004: Дедуплікація статей

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
Opportunity Hunter сканує різні RSS-джерела. Одна й та сама стаття може з'явитися в кількох стрічках (наприклад, TechCrunch та Hacker News). Без дедуплікації в базі накопичуються дублікати, що:
- Засмічує сховище
- Спотворює аналіз (одна проблема враховується кілька разів)
- Генерує повторювані гіпотези

## Рішення

### Підхід
- Додати унікальне обмеження (або перевірку) на поле `link` у метаданих об'єкта.
- Перед збереженням нової статті перевіряти, чи існує об'єкт типу `Article` з таким самим `link`.
- Якщо існує — пропускати збереження (логірувати як дублікат).
- Якщо ні — зберігати як зазвичай.

### Технічна реалізація
- Додати метод у `Storage`: `get_object_by_metadata(key, value, object_type)`.
- Використовувати SQL запит: `SELECT * FROM objects WHERE type=? AND json_extract(metadata, '$.link')=?`.
- Індексувати поле `link` у JSON для швидкості (опціонально, пізніше).

### Альтернативи, що не пройшли
- Складна хеш-функція на основі заголовка + опису — менш надійно, ніж пряме посилання.
- Повне сканування всіх об'єктів у пам'яті — неефективно.

### Наслідки
- База залишається чистою від дублікатів.
- Невелике зниження продуктивності через додатковий запит (але на MVP це непомітно).
- У майбутньому можна додати індекс на JSON поле для прискорення.

## Майбутні плани
- Додати індексацію поля `link` у SQLite (через `json_extract`).
- Розширити дедуплікацію на гіпотези (щоб не генерувати однакові).
