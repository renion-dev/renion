# ADR-005: Оптимізація роботи з LLM (Ollama)

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
Opportunity Hunter збирає до 40 статей за одне сканування і передає їх у LLM для аналізу. Поточний таймаут 60 секунд недостатній для обробки великого промпту, що призводить до помилок і відсутності гіпотез.

## Рішення

### 1. Збільшення таймауту
- Змінити таймаут у `OllamaClient` з 60 до **300 секунд** (5 хвилин).
- Додати можливість конфігурувати таймаут через параметр.

### 2. Обмеження кількості статей
- Передавати в аналізатор тільки **останні 10 унікальних статей** (найновіші за часом публікації).
- Це зменшить навантаження на LLM і прискорить відповідь.

### 3. Скорочення тексту
- Обмежити довжину `summary` до **300 символів** для кожної статті.
- Якщо немає `summary`, використовувати `title` як основний опис.

### 4. Обробка помилок
- Якщо LLM не відповідає протягом таймауту — логувати помилку і продовжувати роботу без генерації гіпотез (але зберігати статті).
- Додати повторну спробу (retry) з експоненційною затримкою (опціонально).

## Альтернативи
- Використання меншої моделі (наприклад, `phi3` або `tinyllama`) — але це знижує якість аналізу.
- Розбиття статей на частини та аналіз окремо — ускладнює логіку групування проблем.

## Наслідки
- Аналіз стає надійнішим, але може займати більше часу (до 5 хвилин).
- Користувач отримує гіпотези, навіть якщо деякі статті не влізли в контекст.
- У майбутньому можна додати пагінацію або асинхронну обробку великих обсягів даних.

## Майбутні плани
- Додати кешування відповідей LLM для однакових промптів.
- Використовувати streaming для поступового виведення результатів.
