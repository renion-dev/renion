# ADR-006: Покращення парсингу JSON-відповідей від Ollama

**Дата:** 2026-07-21  
**Статус:** Прийнято  

## Контекст
Ollama повертає текстові відповіді, які не завжди є валідним JSON. Це призводить до того, що гіпотези не створюються, навіть якщо LLM згенерувала корисну інформацію.

## Рішення

### 1. Гнучкий парсинг JSON
- Шукати JSON-блок у тексті за допомогою регулярного виразу (між `{` та `}`).
- Якщо знайдено — спробувати розпарсити.
- Якщо не вдалося — використовувати резервний метод (розбиття за ключовими словами).

### 2. Резервний метод (fallback)
- Якщо JSON не вдалося отримати, спробувати витягнути структуровані дані за допомогою регулярних виразів:
  - Знайти "Problem:", "MVP:", "Hypothesis:" тощо.
- Повернути словник з ключами `problems` (список) або `raw` з текстом.

### 3. Логування
- Логувати успішні та невдалі спроби парсингу для відстеження проблем.

### 4. Повторна спроба (retry)
- Якщо відповідь не містить JSON, надіслати додатковий запит з більш конкретним промптом (наприклад, "Return only JSON").
- Обмежити кількість спроб до 2.

## Альтернативи
- Використання бібліотеки `json-repair` — зайва залежність, не завжди працює.
- Зміна моделі на більш "слухняну" — не завжди можливо.

## Наслідки
- Більше гіпотез буде згенеровано навіть при "сирих" відповідях від LLM.
- Покращується стабільність системи без зміни моделі.
- Додаткове навантаження на LLM (при retry), але це компенсується обмеженням до 2 спроб.
